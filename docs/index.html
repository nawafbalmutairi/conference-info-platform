This project builds a complete web platform for managing academic conference information end-to-end. The platform serves three audiences: visitors browsing upcoming and past conferences, administrators managing the catalogue through full create/update/delete workflows, and developers integrating with the platform programmatically via a REST API.

The architecture is deliberately classical: PHP backend serving HTML/CSS/JS, a MySQL relational database with normalised tables and foreign-key relationships, session-based authentication for admin actions, and a separate REST API exposing CRUD operations as JSON endpoints with standard HTTP status codes.

The project demonstrates competency across the full web stack — not just a single layer. It covers database schema design, server-side scripting, REST API conventions, frontend integration, and authentication — the foundational skill set for any backend or full-stack analyst-engineer role.

Introduction.

The brief: build a complete conference-management web platform with full CRUD on conferences, programmatic access via REST, and authenticated admin actions. The implicit goal of the module was breadth — every layer of the stack had to be touched, and shortcuts (e.g. using a framework that hides the underlying request lifecycle) would defeat the educational purpose.

The deliberate choice was therefore vanilla PHP + raw MySQL — no Laravel, no Symfony, no Composer-managed framework. The result is closer to how PHP applications actually worked in their dominant era (early-to-mid 2010s) and exposes every part of the stack explicitly.

Research questions.

  • Can a complete CRUD + REST platform be built in vanilla PHP, without modern framework abstractions?
  • What is the right database schema for an academic-conference catalogue?
  • How should web UI and REST API share backend code without coupling them too tightly?

Architecture.

The architecture splits cleanly into four layers, each with a single responsibility:

  • Database (MySQL). Normalised tables for conferences, topics, speakers, venues, and admin users. Foreign-key relationships enforce referential integrity at the database level, not the application level.
  • Backend (PHP). Shared modules in includes/ handle the DB connection, session management, and helper utilities. The web routes and API routes both consume these modules.
  • Web UI (PHP-rendered HTML). Public-facing pages (listing, detail, search) render server-side. Admin pages reuse the same DB modules behind an authentication gate.
  • REST API (PHP JSON endpoints). A parallel set of endpoints under /api/ serves the same data as JSON, with proper HTTP semantics — 200/201/400/401/404/500 status codes, JSON-formatted bodies, and content-type headers.

Why shared modules instead of duplicated logic?

The temptation in a small project is to write the database query once for the web UI and again for the API. The cost shows up later when business logic changes: now two paths must be updated, and they can drift. The shared-module approach pulls all DB access into includes/db.php and includes/helpers.php, so both the web routes and the API routes consume the same canonical functions.

Database Design.

The relational schema is intentionally normalised — denormalisation has a place in production systems, but a coursework project should demonstrate the canonical form first.

Sample schema.

SNIPPET 01 · sql/schema.sql · conferences table
CREATE TABLE conferences ( id INT AUTO_INCREMENT PRIMARY KEY, title VARCHAR(255) NOT NULL, slug VARCHAR(255) NOT NULL UNIQUE, description TEXT, start_date DATE NOT NULL, end_date DATE NOT NULL, venue_id INT, website_url VARCHAR(500), registration_url VARCHAR(500), created_at DATETIME DEFAULT CURRENT_TIMESTAMP, updated_at DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP, FOREIGN KEY (venue_id) REFERENCES venues(id) ON DELETE SET NULL, INDEX idx_start_date (start_date), INDEX idx_slug (slug) );

Three design choices in this schema worth noting: (i) slug column for clean URLs (/conferences/icml-2024), (ii) created_at / updated_at with automatic database-level timestamps, (iii) indices on the columns used in WHERE and ORDER BY — start_date for sorting upcoming conferences, slug for URL lookups.

HTTP status codes.

The API uses standard HTTP semantics: 200 OK for successful retrieval, 201 Created for successful POST, 400 Bad Request for malformed payloads (e.g. invalid date format), 401 Unauthorized for missing or invalid session tokens on protected endpoints, 404 Not Found when an ID doesn't exist, and 500 Internal Server Error for unexpected database failures.

Authentication.

Authentication is session-based: the admin logs in via a form, the server validates credentials against the admin_users table (passwords hashed with password_hash(), PHP's bcrypt wrapper), and a session cookie is set. Subsequent requests carry the session cookie and the server looks up the active session before allowing any admin or API-write action.

!
Session vs JWT. A modern alternative is JWT (stateless tokens). For this project, session-based auth was the right choice because (i) the server-rendered web UI already needs sessions for CSRF protection, and (ii) sessions allow server-side invalidation (e.g. forced logout on password change) which JWTs do not, without extra infrastructure.

Password handling.

Passwords are never stored in plaintext. password_hash($password, PASSWORD_BCRYPT) produces a bcrypt hash including salt; password_verify() validates at login time. This is the standard PHP idiom and avoids the manual hashing/salting bugs that have historically plagued PHP applications.

Discussion · Trade-offs.

Why vanilla PHP over a framework?

A framework (Laravel, Symfony) would have given routing, ORM, middleware, and CLI tooling for free. The cost: the project would teach the framework, not the stack. For a coursework deliverable whose purpose is to demonstrate full-stack competency, the trade-off is the wrong way around. Vanilla PHP forces every line of routing logic, every SQL query, every authentication check to be explicit — which is exactly what an evaluator needs to see.

Why REST over RPC or GraphQL?

REST's resource-orientation maps cleanly onto a CRUD-heavy domain like conferences. GraphQL would shine if the platform had a complex front-end with varying field selections per page; here, the queries are simple and fixed. RPC would have been over-flexible — multiple endpoints doing similar things with no naming convention. REST's HTTP-verb + URL-resource pairing was the right level of abstraction.

The sharedness of /includes/ matters more than it looks

Early versions of this project had separate get_conference() functions in both the web route and the API route. They drifted within two weeks of development — the web version started joining the venue table, the API version returned venue_id only.

Pulling all DB access into includes/helpers.php and consuming the same function from both layers eliminated the drift permanently. This is a small architectural decision but it's the one that determined whether the codebase stayed maintainable or collapsed under inconsistency.

Limitations & Future Work.

  • No automated tests. The project relies on manual QA of the admin and API flows. A future iteration should add PHPUnit tests for the API endpoints and helper functions.
  • No rate limiting on the API. Public GET endpoints would benefit from token-bucket rate limiting to prevent abuse. In a real deployment this would sit at the reverse-proxy layer (Nginx or Cloudflare), not in the PHP application itself.
  • No pagination on list endpoints. GET /api/conferences currently returns the full list. With more than a few hundred conferences, this becomes unwieldy. Adding ?limit= and ?offset= query parameters is the natural fix.
  • No CSRF protection on admin forms. Sessions are used but explicit CSRF tokens are not. A future iteration should add a per-session token validated on every POST/PUT/DELETE.

Conclusion.

The three research questions are answered. RQ1: yes, a complete CRUD + REST platform can be built in vanilla PHP — the result is more explicit and arguably easier to evaluate than a framework version. RQ2: the right schema for an academic-conference catalogue is normalised with M:N relationships for topics and speakers, foreign-key referential integrity, and slug columns for clean URLs. RQ3: web UI and REST API should share all data-access code through a common includes/ layer, with the two route handlers only differing in their output format (HTML vs JSON).

The practical takeaway from this project: the choices that look small early often define the project's maintainability later. Sharing helper functions, normalising the schema before optimising it, and using session-based auth with bcrypt are all simple decisions made early — and they're the decisions that determine whether the codebase is still extensible six months later.
